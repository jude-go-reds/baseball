# Plan: Build the Baseball Card Website

## Context
This is Jude's first software project. The goal is a public web app that generates shareable baseball-card images for any MLB player (all eras), per `PRD.md` in this folder. The folder is currently empty except for the PRD, so this is fully greenfield.

**Decisions locked in (from clarifying questions):**
- **Stack**: Next.js (App Router) + TypeScript, deployed on Vercel.
- **Storage**: static JSON files committed to the repo for the merged historical dataset (no database to manage). Vercel KV may be added later just for the Wikimedia photo-URL cache if needed.
- **Source control**: GitHub, with Vercel auto-deploying from `main`.
- **Scope**: full PRD roadmap (M1–M6), but built milestone-by-milestone — only M1 should be implemented end-to-end first.
- **First-timer**: plan includes setup steps (Node, Vercel, GitHub) Jude hasn't done before.

## Approach: build milestone-by-milestone, ship M1 first

The PRD already defines six milestones. The right move for a first project is to **finish and deploy M1 to a real public Vercel URL before writing any data code**. That proves the entire pipeline (local dev → git → GitHub → Vercel → live URL → image rendering) works on a tiny surface area, and every later milestone is just "add features to a thing that already deploys."

Each milestone below lists the concrete files to create/modify and the tools/libraries involved.

---

### M0 — Local & cloud setup (one-time, do before M1)
1. **Install Node.js 20 LTS** (verify with `node -v`).
2. **Create a GitHub repo** named `baseball` (empty, no README — we'll push from local).
3. **Create a Vercel account** (sign in with GitHub for auto-linking).
4. **Install Vercel CLI** globally: `npm i -g vercel` (used later for previews).
5. **Create CLAUDE.md** in this folder (see *CLAUDE.md content* section below) — this teaches future Claude sessions how to work on this project without re-discovering everything.

### M1 — Skeleton (deploy a hardcoded card)
**Files created:**
- Scaffold via `npx create-next-app@latest baseball --typescript --app --tailwind --eslint` run from `/home/jude/code/` (delete the existing empty `baseball/` first, or scaffold into a temp folder and merge `PRD.md` back in).
- `app/page.tsx` — placeholder home page ("Baseball Cards" + link to a sample card).
- `app/card/[id]/page.tsx` — player page that embeds the card image.
- `app/api/card/[id]/route.tsx` — `@vercel/og` endpoint returning a hardcoded PNG (use one player like Babe Ruth as the demo).
- `lib/cards/templates/modern.tsx` — first JSX card template, hardcoded data.
- `package.json` — add `@vercel/og` dependency.
- `.gitignore`, `README.md` (minimal), `next.config.ts` (default).

**Deploy:** push to GitHub, import the repo into Vercel, confirm `https://<project>.vercel.app/api/card/ruth` returns a PNG.

### M2 — MLB Stats API + search + first real card
**New files:**
- `lib/mlb/client.ts` — thin fetch wrapper around `https://statsapi.mlb.com/api/v1/...` with typed responses for `people`, `stats?stats=career`.
- `lib/mlb/types.ts` — TypeScript types for the player + career-stats shapes we use.
- `lib/players/index.ts` — server-side function `getPlayer(id)` that pulls from MLB API and returns a normalized `Player` object the card template consumes.
- `app/page.tsx` — replace placeholder with a search box (client component using Fuse.js).
- `public/players.json` — prebuilt search index (id, name, aliases, years, team) generated by a build script.
- `scripts/build-search-index.ts` — fetches the MLB roster index and writes `players.json`. Run via `npm run build:index` and commit the output.
- Wire `app/card/[id]/page.tsx` and `app/api/card/[id]/route.tsx` to call `getPlayer(id)` instead of using hardcoded data.

**Position-aware logic** (PRD §4): branch the card template on `player.position` — `'P'` → pitcher card, two-way → both. Keep this in `lib/cards/templates/modern.tsx` for now; split into `Hitter.tsx` / `Pitcher.tsx` / `TwoWay.tsx` if it grows.

### M3 — Photos
**New files:**
- `lib/photos/mlb.ts` — build URL `securea.mlb.com/mlb/images/players/head_shot/<id>.jpg`, HEAD-check it.
- `lib/photos/wikimedia.ts` — query Wikipedia/Wikimedia API by player name; cache resolved URL.
- `lib/photos/index.ts` — `resolvePhoto(player)` falls through MLB → Wikimedia → silhouette placeholder (`public/silhouette.png`).
- Cache resolved URLs: start with an in-memory module-level Map; promote to Vercel KV only if cold-start re-resolution becomes a problem.
- Re-encode photos server-side before passing to `@vercel/og` (PRD §9 risk: 1 MB Edge payload cap).

### M4 — Advanced stats + awards (the historical-data milestone)
**New files:**
- `data/historical/players.json` — merged FanGraphs/B-Ref/Lahman dataset, keyed by MLB player ID, containing: WAR, OPS+, wRC+, ERA+, MVP/CY/AS/GG/SS counts, HoF year, WS rings.
- `scripts/import-historical.ts` — one-off script that reads the source CSVs (place them in `data/sources/` ignored by git), merges and writes `data/historical/players.json`. Runs locally; commit the output JSON.
- `lib/players/index.ts` — extend `getPlayer(id)` to merge historical data on top of MLB API response.
- `lib/cards/templates/modern.tsx` — add honors strip + advanced metric rows.
- **Attribution**: footer link on every player page crediting MLB Stats API, FanGraphs, Baseball-Reference, Lahman, and Wikimedia (PRD §9).

**ToS check before shipping**: re-read FanGraphs and B-Ref free-data terms; verify non-commercial-with-attribution still applies. If not, scope the dataset to Lahman-only (which is permissively licensed) and accept the metric gaps.

### M5 — Style picker
**New files:**
- `lib/cards/templates/vintage.tsx` — 1950s–70s look (cream background, serif type, team logo top-left).
- `lib/cards/templates/minimalist.tsx` — data-viz-forward (sparklines for HR-by-year, etc.).
- Both templates accept the same `Player` prop the modern one does — only presentation differs.
- `app/api/card/[id]/route.tsx` — read `?style=` query param, dispatch to the right template; default to `modern`.
- Style picker UI on `app/card/[id]/page.tsx` (three buttons that swap the `style=` query param).

### M6 — Browse galleries
**New files:**
- `app/browse/hof/page.tsx` — list of Hall of Famers (filter `historical/players.json` by `hofYear != null`).
- `app/browse/team/[team]/page.tsx` — players grouped by primary team.
- `app/browse/era/[era]/page.tsx` — players grouped by debut decade.
- Reuse `players.json` search index — no new data needed.

## Cross-cutting setup (do once, applies to all milestones)

- **Caching** (PRD §7): on `/api/card/[id]/route.tsx`, set `Cache-Control: public, s-maxage=86400, stale-while-revalidate=604800`. Add a `vercel.json` cron entry once active-player refresh becomes relevant (M4 area, not M1).
- **Edge runtime** for the card route: `export const runtime = 'edge'` (required by `@vercel/og`).
- **Font subsetting**: download a TTF subset (Latin only) for any custom fonts; load via `fs.readFileSync` at module scope, not per-request.
- **Disambiguation** (PRD §4): search results must show `name + primary team + years` so the two Ken Griffeys are distinguishable. Build this into `players.json` from M2.

## CLAUDE.md content

Create `/home/jude/code/baseball/CLAUDE.md` with these sections (this is Jude's first project — the file should make a future Claude session productive without spelunking):

1. **Project summary** — one paragraph: "Public web app that generates shareable baseball-card PNGs for any MLB player. Next.js + @vercel/og on Vercel. See PRD.md for full requirements."
2. **Tech stack** — Next.js App Router, TypeScript, Tailwind, `@vercel/og` (Edge), Fuse.js for search, deployed on Vercel from GitHub `main`.
3. **Repo layout** — `app/` routes, `lib/` server logic (mlb client, photos, players, card templates), `data/historical/` committed JSON, `scripts/` build-time data jobs, `public/` static assets including `players.json` search index and silhouette fallback.
4. **How to run locally** — `npm install`, `npm run dev`, open `http://localhost:3000`. Card endpoint at `/api/card/<mlb-player-id>?style=modern`.
5. **How to deploy** — push to `main`; Vercel auto-deploys. Preview URLs come from Vercel CLI (`vercel`) or PRs.
6. **Data sources & attribution rules** — list the four sources from PRD §6 and the rule "every page that displays stats must credit them in the footer."
7. **Conventions** —
   - Card templates live in `lib/cards/templates/`, one file per style, all accept the same `Player` prop.
   - All player data flows through `lib/players/getPlayer(id)` — never call MLB API or read `historical/players.json` directly from a route.
   - Edge runtime constraints: no `fs` reads per-request, no large dependencies — keep payload under 1 MB.
8. **What NOT to do** — don't add a database before it's needed (start with JSON), don't fetch photos at request time without falling back to placeholder, don't ship without source attribution.
9. **Pointer to PRD.md** — "PRD.md in this folder is the source of truth for requirements."
